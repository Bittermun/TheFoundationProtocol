# TFP v2.2 — Security Hardening Notes

## Overview

This document captures the security threat model, hardening decisions, and mitigation
strategies implemented in TFP v2.2. It covers four attack surfaces: content integrity,
identity/Sybil, ledger inflation, and semantic drift.

---

## Threat Model

| Threat | Vector | Severity | Mitigation |
|--------|--------|----------|------------|
| Shard poisoning | Adversarial broadcaster injects corrupt shards | High | SymbolicPreprocessor + RaptorQ redundancy |
| Sybil proof farming | Fake PUF identities mint unlimited credits | Critical | PUFEnclave entropy binding + nonce replay rejection |
| Credit chain tampering | Attacker mutates ledger history | Critical | SHA3-256 append-only hash chain; verify_spend() |
| Semantic drift | Malicious HLT delta diverges model | High | Hash-pinned apply_delta; atomic rollback on mismatch |
| NDN Interest flooding | DoS via Interest storm | Medium | Per-prefix rate limiting in NDN forwarder (future) |
| Replay attacks | Re-submit old signed proofs | High | Per-proof nonce (16 random bytes) in PUFEnclave.sign_posi_proof() |

---

## 1. Credit Ledger — Hash-Chain Integrity

### Design
The `CreditLedger` uses a SHA3-256 append-only chain. Each block commits to:
```
block_n = SHA3-256(block_{n-1} || proof_hash || credits_as_8_bytes_big_endian)
```
Genesis block uses `\x00 * 32` as `block_0`.

### Properties
- **Append-only**: no block can be mutated without invalidating all subsequent blocks
- **Deterministic**: same inputs always produce the same chain hash
- **No external state**: fully in-process; no database dependency at the edge

### Verification
`verify_spend(receipt)` checks `receipt.chain_hash in self._chain` — O(n) scan.
For production, a Bloom filter or sorted set provides O(1) amortized lookup.

### Future hardening
- Export chain as Merkle tree root for remote auditing
- Add threshold signature from multiple PUF enclaves before minting

## 2. PUF Enclave — Identity Binding & Replay Prevention

### Design
```python
puf_entropy   = SHA3-256(seed || b"puf")      # 32 bytes — hardware-bound
rf_fingerprint = MD5(seed || b"rf")            # 16 bytes — channel fingerprint
threshold_sig  = SHA3-512(seed || b"threshold") # 64 bytes — multi-party threshold
```

`sign_posi_proof(proof)` appends a fresh 16-byte nonce before HMAC:
```python
nonce = os.urandom(16)
sig   = HMAC-SHA3-256(seed, proof || nonce) || nonce
```
This ensures identical proofs produce different signatures — replay is rejected.

### Sybil rejection
`verify_identity(identity, expected_seed)` uses `hmac.compare_digest` (constant-time)
to compare `identity.puf_entropy` against the expected derivation. A fake PUF with a
different seed produces a different entropy value with overwhelming probability
(collision probability ≈ 2⁻²⁵⁶).

### Production upgrade path
- Replace `seed` with TRNG output latched at manufacturing time and stored in OTP
- Bind `rf_fingerprint` to real I/Q samples via libiio SDR pipeline (see `sdr_pipeline.grc`)

## 3. Symbolic Preprocessor — Content Validation

### Rules enforced (v2.2)
| Rule | Check | Confidence impact |
|------|-------|-------------------|
| Required fields present | `{task_type, params_hash, difficulty}` ⊆ recipe | -∞ (hard fail) |
| Byte size limit | `len(raw_bytes) ≤ 2048` | -∞ (hard fail) |
| `difficulty` ≥ 0 | Type + range check | -0.5 on fail |
| `task_type` non-empty string | Type + strip check | -∞ on fail |
| `params_hash` length = 64 | SHA3-256 hex length | +0.15 on pass |

### Poison detection
Poisoned recipes are rejected at `validate()` before any RaptorQ decode attempt.
This prevents compute waste and stops payload-injection via crafted shard metadata.

### Confidence scoring
Confidence ∈ [0.0, 1.0]. Values < 0.5 trigger quarantine in future versions.
Currently: valid = True requires confidence ≥ baseline (all hard rules pass).

## 4. Hierarchical Lexicon Tree — Atomic Delta Application

### Hash-pinned updates
```python
def apply_delta(self, base_pin_hash: str, delta: bytes) -> bool:
    if self._base_hash != base_pin_hash:
        return False   # mismatch → no write, state unchanged
    ...
```
Any delta that references the wrong base is silently rejected. No partial state is written.

### Rollback guarantee
On exception during delta application, `backup = copy.deepcopy(self._state)` is restored.
This ensures the tree is never left in a partially-applied state.

### Semantic drift prevention
By requiring the `base_pin_hash` to match exactly, an adversary cannot inject a forked
delta branch without compromising the hash. Drift divergence is bounded to zero for any
node that correctly implements the protocol.

## 5. RaptorQ Fountain Erasure Code — Shard Integrity

### Redundancy
`encode(data, redundancy=0.05)` generates `k + ceil(k * 0.05) + 1` shards.
This means up to ~5% of shards can be lost or corrupted without affecting reconstruction.

### Systematic design
Source shards (indices 0..k-1) are stored verbatim. If all source shards arrive,
decode is O(k) concatenation — no Gaussian elimination needed.

### Tamper detection
Each shard header carries `(orig_len, k, index)`. Mismatched headers cause decode to
fall back to legacy mode or raise. Future versions will add a per-shard HMAC.

## 6. Asymmetric Uplink — Channel Selection Security

The `AsymmetricUplinkRouter` weights `(latency, energy, drop_rate)` with a sum-to-1.0
constraint. The exponential backoff on high drop rates (`drop_rate > 0.5 → backoff = 2^(drop*2)`)
prevents an adversary from advertising a high-drop channel as low-cost.

**Input validation**: weights are checked at construction time:
```python
if abs(w_latency + w_energy + w_drop - 1.0) > 1e-6:
    raise ValueError(...)
```
