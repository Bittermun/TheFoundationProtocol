## 1. What Changed in v3.0

| Area | v2.12 | v3.0 |
|------|-------|------|
| **Compute proof** | Hash a string, claim credits | Real PoW: HASH_PREIMAGE / MATRIX_VERIFY / CONTENT_VERIFY |
| **Credit minting** | Immediate on earn call | Only after HABP consensus (3/5 devices) |
| **Supply cap** | None | Hard cap: 21,000,000 credits |
| **Consensus survival** | Lost on restart | HABP proofs persisted to SQLite `task_results` table |
| **Metrics restart** | Reset to 0 | Seeded from SQLite on startup |
| **Device visibility** | None | `GET /api/devices` leaderboard + `GET /api/device/{id}` |
| **`/api/device/{id}` query** | Loads all rows, O(N) | Direct SQL, O(1) via `device_stats()` |
| **Task lifecycle** | No expiry cleanup | Auto-reaper marks expired tasks failed, replenishes pool |
| **Pool maintenance** | On-demand only | Background thread reaps + replenishes every 30 s |
| **Content listing** | All items, no limit | Paginated: `?limit=N&offset=N`, returns `total` |
| **Content multi-tag search** | Single tag only | `?tags=a,b,c` union — matches any listed tag |
| **DB concurrency** | Default journal mode | SQLite WAL mode — concurrent reads, no lock errors |
| **total_enrolled** | Capped by limit param | True DB count, always accurate |
| **Result submission rate limit** | None | Per-device `TFP_RESULT_RATE_MAX/WINDOW` (default 30/60s) |
| **`/api/tasks` response** | `tasks` list only | + `open_count` (total pool size, independent of `limit`) |
| **CLI search** | JSON dump, no pagination | Table output, `--limit/--offset/--tags` flags |
| **CLI** | `tfp publish`, `tfp get` | + `tfp join`, `tfp tasks`, `tfp leaderboard`, `tfp search` |
| **Admin dashboard** | Tasks + supply bar | + Device leaderboard table |
| **Docker** | Port mapping only | + HEALTHCHECK, persistent named volume, env vars |
| **Test suite** | 408 passing | **491 passing, 1 warning** |

**Economic flywheel:**
```
Device enrolls → polls /api/tasks → executes task locally (real PoW)

  → credits spent on /api/get/{hash} (content access)
```

## 4. API Reference

### 4.1 Device Enrollment & Auth

#### `POST /api/enroll`
Enroll a new device. Creates a device identity backed by a PUF entropy secret.

```json
// Request
{ "device_id": "my-laptop", "puf_entropy_hex": "<64 hex chars>" }

// Response 200
{ "device_id": "my-laptop", "enrolled": true }
```

**Auth for protected endpoints:** Every call to `/api/publish`, `/api/earn`, and
`/api/task/{id}/result` requires the header:
```
X-Device-Sig: HMAC-SHA-256(puf_entropy_bytes, "device_id:message_field")
```
Where `message_field` is `title` for publish, `task_id` for earn, and
`device_id:task_id` for task result submission.

### 4.2 Compute Task API

#### `POST /api/task`
Create a new compute task. Server generates a solvable problem + known answer.

```json
// Request (optional — defaults shown)
{ "task_type": "hash_preimage", "difficulty": 3, "seed_hex": "" }

// Response 200
{
  "task_id": "a1b2c3d4",
  "task_type": "hash_preimage",
  "difficulty": 3,
  "credit_reward": 12,
  "deadline": 1234567890.0,
  "input_data_hex": "deadbeef...",
  "expected_output_hash": "3f8a12..."
}
```

`task_type` must be one of: `hash_preimage`, `matrix_verify`, `content_verify`.

---

#### `GET /api/tasks`
List open tasks available for execution.

```
GET /api/tasks?limit=20
```

```json
{
  "tasks": [
    {
      "task_id": "a1b2c3d4",
      "task_type": "hash_preimage",
      "difficulty": 3,
      "credit_reward": 12,
      "deadline": 1234567890.0,
      "time_left_s": 3542
    }
  ],
  "open_count": 5
}
```

`open_count` is the **total** number of open tasks in the pool — independent of `limit`.
Use it to determine whether more tasks are available than returned.

Expired tasks are reaped automatically before the list is returned.

---

#### `GET /api/task/{task_id}`
Get full task spec including input data.

```json
{
  "task_id": "a1b2c3d4",
  "task_type": "hash_preimage",
  "difficulty": 3,
  "status": "verifying",
  "credit_reward": 12,
  "created_at": 1234567800.0,
  "deadline": 1234567890.0,
  "input_data_hex": "deadbeef...",
  "expected_output_hash": "3f8a12..."
}
```

---

#### `POST /api/task/{task_id}/result`
Submit a computed result. Requires device auth header.

```json
// Request
{
  "device_id": "my-laptop",
  "output_hash": "3f8a12...",    // SHA3-256 hex of your computed result
  "exec_time_s": 0.14,
  "has_tee": false
}

// Response — pending consensus
{
  "status": "pending_consensus",
  "verified": false,
  "credits_earned": 0,
  "consensus_needed": 2,
  "proofs_received": 1
}

// Response — consensus reached (3rd matching device)
{
  "status": "verified",
  "verified": true,
  "credits_earned": 12,
  "consensus_needed": 0,
  "matching_devices": ["device-a", "device-b", "my-laptop"],
  "confidence": 1.0
}
```

When `verified=true`, credits are automatically minted to your device's ledger.

**Rate limiting:** To prevent a single device from flooding the HABP verifier, result
submissions are rate-limited per device. Default: 30 submissions per 60-second window.
Excess requests return `HTTP 429`. Configure via env vars:
```
TFP_RESULT_RATE_MAX=30    # max submissions per window
TFP_RESULT_RATE_WINDOW=60  # window size in seconds
```

Publish content (text). Requires auth. Costs nothing; earns the network.

```json
{
  "title": "My Article",
  "text": "Body text here…",
  "tags": ["technology", "decentralized"],
  "device_id": "my-laptop"

```

Search content by tag with optional pagination. Returns `root_hash`, `title`, `tags`, plus

**Multi-tag union search:** Use `?tags=tag1,tag2,tag3` to return items matching **any** listed tag.

```json

  "total": 42,
  "limit": 20,
  "offset": 0
}
```

Defaults: `limit=50`, `offset=0`, `limit` max is 500.

#### `GET /api/get/{root_hash}?device_id=my-laptop`
Retrieve content. **Costs 1 credit** (deducted from device's earned receipts).
Returns `402` if no credits available.

## 5. Task Types & Proof-of-Work

Three task types are supported, each providing a different compute profile:

### `hash_preimage` (PoW)
Find a nonce such that `SHA3-256(seed + nonce)` has `difficulty` leading zero bits.
Scales quadratically — difficulty 3 ≈ 0.1s, difficulty 10 ≈ minutes on a modern CPU.

```python
from tfp_client.lib.compute.task_executor import execute_task, generate_hash_preimage_task

spec = generate_hash_preimage_task(task_id="t1", difficulty=3, seed=b"myseed")
result = execute_task(spec)
print(result.output_hash)  # SHA3-256 hex
```

### `matrix_verify`
Verify C = A × B mod p for randomly generated matrices.
Deterministic output — all devices with the same spec produce identical `output_hash`.

### `content_verify`
Iterated SHA3-256 over input content (N rounds = difficulty × 1000).
Pure hash work; useful for verifying content integrity across the network.

### TaskSpec serialisation
```python
spec.to_dict()  # → JSON-serialisable dict
TaskSpec.from_dict(d)  # → TaskSpec
spec.input_data       # bytes — what devices need to execute
spec.expected_output_hash  # hex — server-generated ground truth
```

## 6. HABP Consensus Protocol

**Hardware-Agnostic Benchmark Proof (HABP)** prevents a single device from
self-minting credits by requiring agreement from multiple independent devices.

### Protocol
1. Device executes task locally → produces `output_hash = SHA3-256(result)`
2. Device submits `output_hash` to `/api/task/{id}/result`
3. Server stores proof in `task_results` (SQLite) — survives restarts
4. When ≥3 proofs share the same `output_hash`:
   - Consensus reached → `verified = true`
   - Credits calculated via `CreditFormula(difficulty, consensus_confidence)`
   - Credits minted to all devices in the matching group

### Parameters
| Parameter | Default | Description |
|-----------|---------|-------------|
| `consensus_threshold` | 3 | Minimum matching proofs |
| `redundancy_factor` | 5 | Target proofs per task |

### Restart safety
On server startup, `TaskStore._rebuild_habp_from_db()` replays all persisted
proofs from `task_results` for tasks still in `verifying` state. Consensus
accumulates across server restarts with no proof loss.

### Credit weights
| Condition | Weight |
|-----------|--------|
| 3/5 consensus | 1.0 × base |
| 4/5 consensus | 1.1 × base |
| 5/5 consensus | 1.2 × base |
| TEE attestation | 1.5 × base (immediate, no consensus needed) |

## 7. Credit Economics & Supply Cap

### Hard Cap
```python
MAX_SUPPLY = 21_000_000  # from tfp_client/lib/credit/ledger.py
```
`CreditLedger.mint()` raises `SupplyCapError` if `network_total + amount > MAX_SUPPLY`.
`TaskStore.increment_total_minted()` enforces the same check atomically in SQLite.

### Credit formula
Credits per task ≈ `base_credits × difficulty_multiplier × consensus_confidence × hardware_trust`.
Minimum reward: 1 credit. Maximum single-task reward: ~50 credits at difficulty 10 + TEE.

### Supply tracking
- Network total persisted in `supply_ledger` SQLite table (single row, `id=1`)
- `CreditLedger.set_network_total_minted()` is called before every mint
- Prometheus counter `tfp_credits_minted_total` is seeded from DB on startup

### Chain integrity
Each credit receipt contains `chain_hash = SHA3-256(prev_hash + proof_bytes)`.
The chain is Merkle-auditable: `CreditLedger.merkle_root()` changes on every mint.

## 8. CLI: `tfp join`

The `tfp join` command implements the full device participation loop:

```bash
tfp --api http://localhost:8000 join \
  --device-id my-laptop \
  --interval 5
```

**What it does:**
1. **Enroll** — generates PUF entropy, calls `/api/enroll`, persists identity to `~/.tfp/identity.json`
2. **Poll** — calls `GET /api/tasks` every `--interval` seconds
3. **Execute** — runs the task locally using `TaskExecutor`
4. **Submit** — calls `POST /api/task/{id}/result` with auth header
5. **Report** — prints consensus status and credits earned total
6. **Loop** — repeats until interrupted (Ctrl+C)

**Identity persistence:**
```json
// ~/.tfp/identity.json
{
  "device_id": "my-laptop",
  "puf_entropy_hex": "abc123..."
}
```

**Other CLI commands:**
```bash
python -m tfp_cli.main publish "My Title" "Content text…" --tags tech,demo \
  --device-id my-laptop
python -m tfp_cli.main get <root_hash> --device-id my-laptop

# Search with pagination and table output
python -m tfp_cli.main search                             # all content
python -m tfp_cli.main search --tag technology            # single tag
python -m tfp_cli.main search --tags science,technology   # union (any tag)
python -m tfp_cli.main search --limit 10 --offset 20     # paginate

# Tasks and devices
python -m tfp_cli.main tasks --limit 10
python -m tfp_cli.main leaderboard --limit 20
```

## 9. SQLite Persistence & Restart Safety

All state is persisted to a single SQLite database (`TFP_DB_PATH`, default: `pib.db`).

| Table | Contents |
|-------|----------|
| `content` | Published items (root_hash, title, tags, data) |
| `devices` | Enrolled device IDs + PUF entropy |
| `credit_ledger` | Per-device balance + chain + unspent receipts |
| `earn_log` | Deduplication log (device_id, task_id) — prevents replay |
| `tasks` | Task lifecycle (open → verifying → completed/failed) |
| `task_results` | Per-device results (unique on device_id+task_id) |
| `supply_ledger` | Single-row global minted total |

**Restart safety guarantees:**
- Content, device enrollments, credits, and task results survive restarts
- HABPVerifier in-memory state is rebuilt from `task_results` on startup
- Prometheus metrics seeded from DB on startup
- Expired tasks auto-reaped on next `GET /api/tasks`

**SQLite WAL mode:**
The demo server enables WAL (Write-Ahead Logging) journal mode automatically for
file-based databases. WAL allows concurrent reads while a write is in progress,
eliminating "database is locked" errors common in multi-worker deployments.

**Background maintenance thread:**
A daemon thread runs every 30 seconds to:
1. Call `reap_expired_tasks()` — marks open/verifying tasks past deadline as `failed`
2. Call `_preseed_tasks()` — replenishes the pool to minimum size (5 tasks)

This means the pool stays healthy even when no HTTP traffic is hitting `/api/tasks`.

## 13. Security Model

### Device authentication
- Every mutating endpoint requires `X-Device-Sig: HMAC-SHA-256(puf_entropy, message)`
- Unenrolled device → `401 Unauthorized`
- `_verify_device_sig` uses `hmac.compare_digest` (constant-time) to prevent timing attacks

### Anti-replay
- `EarnLog` SQLite table with UNIQUE constraint on `(device_id, task_id)` — returns `409` on duplicate
- `task_results` table has UNIQUE on `(task_id, device_id)` — `INSERT OR IGNORE` prevents double-submit

### Supply cap
- `CreditLedger.mint()` raises `SupplyCapError` if network total would exceed 21M
- `TaskStore.increment_total_minted()` uses a threading lock + SQLite transaction
- Both checks are enforced independently (defence in depth)

### Rate limiting
- Sliding-window token bucket per device on `/api/earn` (default: 10 calls/60s)
- Sliding-window token bucket per device on `POST /api/task/{id}/result` (default: 30/60s)
- Both return `429 Too Many Requests` on breach; limits configurable via env vars

### Sybil resistance
- HABP consensus requires 3 distinct `device_id` values submitting matching results
- In production: combine with PUF/TEE attestation for hardware-bound identity

### Content integrity
- Every published item stored by `root_hash = SHA3-256(content_bytes)`
- Content retrieved via `/api/get/{hash}` is verified against the stored hash
- Merkle audit trail on credit chain prevents ledger tampering

### Threat model updates (v3.0)
| Threat | Mitigation |
|--------|-----------|
| Self-mint (1 device submits 3 fake proofs) | `task_results` UNIQUE(task_id, device_id) — each device can submit exactly one proof per task |
| Sybil devices submitting wrong output_hash | Wrong hashes don't reach consensus threshold; task remains in verifying |
| Supply exhaustion attack | 21M hard cap; supply check before every mint |
| Replay after restart | Task results persisted; HABP rebuilt from DB on restart |
| Metric manipulation | Counters seeded from SQLite; not user-controllable |

### Known limitations (single-node demo)
The demo server has security properties that hold for single-process deployments but require
attention at scale:

- **Rate limiters are in-memory** — `_RateLimiter` is per-process and resets on restart. For
  multi-node or multi-worker deployments, replace with a Redis-backed implementation.
- **Auto-mint failure is swallowed** — the `except Exception` block around the auto-mint in
  `submit_task_result` logs a warning but does not surface the failure to the caller. Monitor
  `tfp_credits_minted_total` to detect discrepancies.
- **Sybil resistance is ID-only in the demo** — HABP consensus relies on `device_id` uniqueness.
  In production, couple enrollment to hardware PUF/TEE attestation.

For the full security verification checklist and maintenance policy see [SECURITY.md](SECURITY.md).

## 15. Extension Guide

### Adding a new task type

1. Add a new `TaskType` enum value in `tfp_client/lib/compute/task_executor.py`
2. Implement `generate_<type>_task(task_id, difficulty, **kwargs) → TaskSpec`
3. Implement the execution branch in `execute_task(spec) → TaskResult`
4. Implement the verification branch in `verify_result(spec, result) → bool`
5. Register in `TaskStore._GENERATORS` in `tfp_demo/server.py`
6. Add tests in `tests/test_grand_completion.py` under `TestTaskExecutorUnit`

### Adding a new credit formula

Override `CreditFormula.calculate_credits()` in `tfp_client/lib/compute/credit_formula.py`.
The method receives: `difficulty`, `hardware_trust`, `uptime_hours`,
`verification_confidence`, `is_charging`.

### Plugin SDK

The `PluginSDK` class in `tfp_client/lib/plugins/sdk.py` allows third-party
extensions to hook into content publish/retrieve and the credit earn/spend lifecycle.
See the existing bridge plugins (`ipfs_bridge.py`, `nostr_bridge.py`) for examples.

## 17. Documentation Maintenance Policy

### Single source of truth

| Doc type | Canonical location |
|----------|--------------------|
| **Integration guide** (current) | `docs/v3.0-integration-guide.md` (this file) |
| **Security model & checklist** | `docs/SECURITY.md` |

| **Project README** | `/README.md` (top-level, pointers only) |
| **Historical strategy docs** | `docs/archive/` in root (read-only) |

### Per-release update checklist

Every PR that bumps the version, adds/removes an endpoint, changes security behaviour, or
modifies the test suite **must** update:

1. `pyproject.toml` — `version` field
2. Root `README.md` — version badge, test-count badge, "Current Status" section, Quick Start test command
3. This guide — "What Changed" table (Section 1), Section 16 test count
4. `SECURITY.md` — if any security property changes or a known limitation is resolved

### Stale-claim rule

Any claim of the form `"N tests passing"` in documentation must match the output of:

```bash
cd tfp-foundation-protocol
TFP_DB_PATH=:memory: PYTHONPATH=. python -m pytest tests/ -q --tb=no
```

The CI `docs-freshness` job (`.github/workflows/ci.yml`) asserts this automatically on every push.

### In-memory rate limiter caveat

All deployment documentation must note that `_RateLimiter` is per-process and does not persist
across restarts. The rate-limiting security claim applies **only to single-process deployments**.
Multi-node operators must substitute a shared backing store before making equivalent claims.
